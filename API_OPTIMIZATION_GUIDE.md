# API レスポンス速度最適化ガイド

このドキュメントは、Rapitasバックエンドに実装された最新の応答速度改善技術について説明します。

## 実装された最適化技術

### 1. パフォーマンス最適化ミドルウェア (`middleware/performance.ts`)

- **レスポンス圧縮**: Gzip/Brotli圧縮のサポート
- **メモリキャッシュ**: LRUキャッシュによる高速レスポンス
- **ETag対応**: 条件付きリクエストによる帯域幅削減
- **パフォーマンスモニタリング**: レスポンスタイムとDB クエリの追跡
- **レート制限**: スライディングウィンドウ方式での過負荷防止

### 2. データベースクエリ最適化 (`utils/prisma-optimization.ts`)

- **選択的フィールド読み込み**: 必要なフィールドのみ取得
- **バッチ処理**: 大量データの効率的な処理
- **並列クエリ実行**: 独立したクエリの同時実行
- **カーソルベースページネーション**: スケーラブルなページング
- **自動リトライロジック**: 一時的なエラーへの対応
- **DataLoader パターン**: N+1問題の解決

### 3. キャッシングサービス (`services/cache-service.ts`)

- **マルチレベルキャッシュ**: メモリ（L1）とRedis（L2）の階層化
- **タグベース無効化**: 関連データの一括無効化
- **キャッシュウォーミング**: 起動時の事前読み込み
- **統計情報**: ヒット率の監視とチューニング

### 4. 最適化されたバッチAPI (`routes/batch-v2.ts`)

- **並列処理**: 複数リクエストの同時実行
- **インテリジェントキャッシング**: リクエスト結果の再利用
- **インクリメンタル更新**: 差分データのみの取得
- **実行統計**: パフォーマンスメトリクスの提供

### 5. WebSocketリアルタイム通信 (`services/websocket-service.ts`)

- **双方向通信**: サーバープッシュによる即時更新
- **ルームベース配信**: 効率的なメッセージルーティング
- **ヘルスチェック**: 接続状態の自動管理
- **リアルタイムコラボレーション**: カーソル共有、タイピング状態

### 6. 最適化されたAPIクライアント (`api-client-optimized.ts`)

- **自動バッチング**: 複数リクエストの自動統合
- **ローカルキャッシュ**: クライアント側でのデータ保持
- **WebSocket統合**: リアルタイム更新の受信
- **自動リトライ**: ネットワークエラーへの対応

## 必要なパッケージのインストール

バックエンドに以下のパッケージをインストールしてください：

```bash
cd rapitas-backend

# キャッシュ関連
bun add lru-cache
bun add ioredis

# WebSocket
bun add @elysiajs/websocket

# 型定義（開発依存）
bun add -d @types/lru-cache
```

フロントエンドに以下のパッケージをインストールしてください：

```bash
cd rapitas-frontend

# WebSocketクライアント
npm install socket.io-client
npm install -D @types/socket.io-client
```

## 使用方法

### 1. 最適化されたバックエンドの起動

現在の`index.ts`を`index-optimized.ts`に置き換えるか、新しいエントリーポイントとして使用：

```bash
# 開発環境
bun run --watch index-optimized.ts

# 本番環境
bun run index-optimized.ts
```

### 2. 環境変数の設定

```env
# Redis URL（本番環境用）
REDIS_URL=redis://localhost:6379

# Node環境
NODE_ENV=production
```

### 3. フロントエンドでの使用

```typescript
import { apiClient } from '@/lib/api-client-optimized';

// 初期化
apiClient.connectWebSocket();

// バッチリクエスト（自動的にバッチ処理される）
const [task1, task2, task3] = await Promise.all([
  apiClient.getTask('1'),
  apiClient.getTask('2'),
  apiClient.getTask('3')
]);

// リアルタイム更新の購読
apiClient.subscribe('task', '1');
apiClient.on('task-updated', (data) => {
  console.log('Task updated:', data);
});
```

## パフォーマンス改善の期待値

1. **レスポンスタイム**: 50-70%の削減（キャッシュヒット時）
2. **データベースクエリ**: 30-50%の削減（バッチ処理とDataLoader）
3. **ネットワーク帯域**: 40-60%の削減（圧縮とバッチング）
4. **リアルタイム性**: WebSocketによる即時更新（<100ms）
5. **スケーラビリティ**: 同時接続数の3-5倍向上

## ベストプラクティス

1. **キャッシュ戦略**
   - 頻繁に変更されるデータ: 短いTTL（1-5分）
   - 静的なマスタデータ: 長いTTL（1時間以上）
   - ユーザー固有データ: タグベース無効化

2. **バッチリクエスト**
   - 関連するデータは一度に取得
   - 最大10-20リクエストを1バッチに
   - 優先度の高いリクエストは個別送信

3. **WebSocket使用**
   - 画面表示中のデータのみ購読
   - 画面遷移時は購読解除
   - 接続状態を監視してフォールバック

4. **エラーハンドリング**
   - キャッシュミスは通常のフローで処理
   - WebSocket切断時は自動再接続
   - バッチエラーは個別リトライ

## トラブルシューティング

### キャッシュが効かない場合
- Redisの接続確認
- TTL設定の見直し
- キャッシュキーの一意性確認

### WebSocketが繋がらない場合
- ファイアウォール設定確認
- プロキシ設定の確認
- フォールバックトランスポートの設定

### パフォーマンスが改善しない場合
- モニタリングメトリクスの確認
- データベースインデックスの最適化
- 並列度の調整

## まとめ

これらの最適化により、Rapitasは以下を実現します：

- **高速レスポンス**: インメモリキャッシュとバッチ処理
- **効率的な通信**: 圧縮とインクリメンタル更新
- **リアルタイム性**: WebSocketによる即時反映
- **スケーラビリティ**: 負荷分散とレート制限
- **開発者体験**: 透過的なAPIクライアント

実装は段階的に導入可能で、既存のコードへの影響を最小限に抑えています。

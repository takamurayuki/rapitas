# 開発環境とアーキテクチャ

## プロジェクト構成
- **rapitas-frontend/**: Next.js 16 + Tailwind CSS v4 + TypeScript（ポート3000）
- **rapitas-backend/**: Elysia + Bun + Prisma ORM + PostgreSQL（ポート3001）
- **rapitas-desktop/**: Tauri 2.x デスクトップアプリ

## 開発サーバーの起動と管理
- 開発サーバーは `rapitas-desktop/scripts/dev.js` が一括管理する。
- **dev.js が起動時に自動実行する処理:**
  1. ポート3001/3000の競合検出・解消（前回クラッシュ時のゾンビプロセス対策）
  2. `prisma db push --skip-generate`（DBスキーマの同期）
  3. `bun run db:generate`（Prisma Clientの生成）
  4. バックエンド・フロントエンドの起動

## 重要な制約事項

### バックエンドサーバーの保護
- **バックエンドサーバー（bunプロセス、ポート3001）を絶対に停止・終了（taskkill等）しないこと。**
  - このサーバーはエージェント自身がAPI通信に使用しているため、停止するとエージェントの実行自体が中断される。
  - 他のNode.js/bunプロセスをkillする前に、それがバックエンドサーバーでないことを必ず確認すること。

### Prismaコマンドについて
- **`prisma generate` や `prisma db push` をエージェントが手動で実行する必要はない。**
  - これらはdev.jsがサーバー起動時に自動実行済み。
  - `schema.prisma` を変更した場合でも、エージェントがこれらのコマンドを実行してはならない。サーバーの再起動が必要になり、エージェント自身の通信が切断される。
  - スキーマ変更後の反映はユーザーがサーバーを再起動することで自動的に行われる。

# GitHub 運用ルール

- **URL:** `https://github.com/taka-y-0820`
- **ツール:** GitHub操作（接続・起票・閲覧）には必ず `gh` コマンドを使用すること。
- **Issueベース開発:**
  - 修正や機能追加はIssueの内容に基づいて実施する。
  - Issue起票時は、AIが単独で実装・完結できるレベルまで詳細な技術仕様を記述すること。
  - Issueの担当者は、進捗状況を定期的に更新し、完了後は必ずレビューを依頼すること。
- **Pull Request (PR) の運用:**
  - PRは必ずIssueに紐づけて作成すること。
  - PRのタイトルは「[Issue番号] 簡潔な変更内容」とする。
  - PRの説明には、変更内容の概要、関連Issue、テスト手順を明記すること。
  - レビューアはコードの品質、動作確認、セキュリティ面を重点的にチェックすること。
  - マージはレビュー完了後に行い、マージコミットには「Merge pull request #PR番号 from ブランチ名」を使用すること。
- **ブランチ戦略:**
  - `main` ブランチは常にデプロイ可能な状態を保つこと。
  - 新機能や修正は `feature/` または `bugfix/` プレフィックスを持つブランチで作業すること。
  - ブランチ名は「プレフィックス/Issue番号-簡潔な説明」とすること。
- **コードレビュー:**
  - すべてのコード変更は少なくとも1人のチームメンバーによるレビューを受けること。
  - レビューコメントには具体的な改善点や質問を含めること。
  - 承認後、必要に応じて修正を行い、再レビューを依頼すること。
- **コミットメッセージ規約:**
  - コミットメッセージは「タイプ: 簡潔な説明」の形式を使用すること。
  - タイプには `feat` (新機能), `fix` (バグ修正), `docs` (ドキュメント), `style` (フォーマット), `refactor` (リファクタリング), `test` (テスト追加・修正), `chore` (その他) を使用すること。
  - メッセージは英語で記述し、50文字以内に収めることを推奨する。
- **リリース管理:**
  - リリースは `release/` プレフィックスを持つブランチ
  - リリースブランチは安定版のコードを含み、必要に応じてホットフィックスを適用する。
  - リリースノートには新機能、修正点、既知の問題を明記すること。
  - リリース後、`main` ブランチにマージし、タグを付与すること。
  - デプロイは自動化されたCI/CDパイプラインを使用して実施すること。
- **CI/CDパイプライン:**
  - すべてのプルリクエストは自動テストとビルドプロセスを通過する必要がある。
  - CI/CDパイプラインの設定はコードベースに含め、変更時にはドキュメントを更新すること。
  - デプロイ前に必ずステージング環境での動作確認を行うこと。
- **セキュリティ対策:**
  - 機密情報（APIキー、パスワードなど）はコードベースに含めないこと。環境変数やシークレット管理ツールを使用すること。
  - 依存関係の脆弱性は定期的にチェックし、必要に応じてアップデートを行うこと。
  - セキュリティインシデントが発生した場合は、速やかにチームに報告し、対応策を講じること。
- **ドキュメント管理:**
  - プロジェクトのREADME、CONTRIBUTINGガイドライン、コードド
    キュメントは常に最新の状態に保つこと。
  - 重要な設計決定やアーキテクチャの変更はドキュメントに記録すること。
  - ドキュメントはMarkdown形式で記述し、リポジトリ内に保存すること。
- **コミュニケーション:**
  - 重要な変更やリリース情報はチーム全体に共有すること。
  - GitHubのIssueやPRでのコミュニケーションは明確かつ丁寧に行うこと。
  - 定期的なミーティングを開催し、進捗状況や課題を共有すること。
- **バックアップとリカバリ:**
  - 重要なデータやコードベースは定期的にバックアップを取得すること。
  - バックアップの保存場所とリカバリ手順をドキュメント化し、チーム全体で共有すること。
- **ライセンス遵守:**
  - 使用しているすべてのオープンソースライブラリやツールのライセンスを確認し、遵守すること。
  - プロジェクトに新しい依存関係を追加する際は、ライセンスの互換性を確認すること。
- **定期的なルール見直し:**
  - GitHub運用ルールは定期的に見直し、必要に応じて更新すること。
  - ルールの変更はチーム全体で合意を得た上で実施し、ドキュメントに反映すること。

# コード品質基準:

- **コードスタイル:**
  - 一貫したコードスタイルを維持するために、プロジェクトで使用するコーディング規約を定めること。
  - 自動フォーマッター（例: Prettier, ESLint）を導入し、コードの整形を自動化すること。
  - typeScriptやPythonなどの静的型付け言語を使用し、型安全性を確保すること。any型の使用は最小限に抑えること。
- **コードの可読性:**
  - 意味のある変数名、関数名、クラス名を使用し、コードの意図が明確になるようにすること。
  - 関数やメソッドは単一の責任を持つように設計し、長すぎる関数は分割すること。
  - コメントは必要最低限に留め、コード自体が自己説明的であることを目指すこと。
- **命名規則:**
  - 一貫した命名規則を採用し、プロジェクト全体で統一すること（例: キャメルケース、スネークケースなど）。
  - クラス名は大文字で始め、関数名や変数名は小文字で始めること。
  - 定数はすべて大文字でアンダースコアで区切ること。
  - コンポーネントのファイル名はケバブケースを使用すること。

- **テストカバレッジ:**
  - すべての新機能およびバグ修正には、対応するユニットテストおよび統合テストを作成すること。
  - テストカバレッジは最低でも80%以上を維持し、定期的にカバレッジレポートを確認すること。
- **パフォーマンス最適化:**
  - コードのパフォーマンスを定期的に評価し、必要に応じて最適化を行うこと。
  - パフォーマンスに影響を与える変更は、ベンチマークテストを実施し、その結果をドキュメント化すること。
- **セキュリティベストプラクティス:**
  - セキュアコーディングのベストプラクティスを遵守し、潜在的なセキュリティ脆弱性を防止すること。
  - 定期的にコードレビューを実施し、セキュリティ上の問題点を早期に発見・修正すること。
- **ドキュメント化:**
  - すべての主要なコード変更には、適切なドキュメントを添付し、変更内容や使用方法を明確にすること。
  - APIドキュメントは常に最新の状態に保ち、新しいエンドポイントや変更点を反映すること。
- **継続的インテグレーション/デリバリー (CI/CD):**
  - CI/CDパイプラインを活用し、コードのビルド、テスト、デプロイを自動化すること。
  - すべてのコード変更はCI/CDパイプラインを通過し、品質基準を満たしていることを確認すること。
- **コードレビュー文化:**
  - チーム全体でコードレビューを積極的に行い、知識の共有とコード品質の向上を図ること。
  - 建設的なフィードバックを提供し、レビューアと開発者の双方が学び合う文化を促進すること。
- **技術的負債の管理:**
  - 技術的負債を定期的に評価し、優先順位を付けて解消する計画を立てること。
  - 技術的負債の解消は、スプリント計画やリリースサイクルに組み込み、継続的に改善を図ること。
- **継続的学習と改善:**
  - チームメンバーは最新の技術動向やベストプラクティスを学び続け、コード品質の向上に努めること。
  - 定期的な振り返りを実施し、コード品質基準の効果を評価し、必要に応じて改善策を講じること。

# コミットメッセージ規約

- **フォーマット:**
  - コミットメッセージは以下の形式を遵守すること:

    ```
    <タイプ>(<スコープ>): <簡潔な説明>

    <詳細な説明（オプション）>

    <関連するIssue番号（オプション）>
    ```

  - `<タイプ>`: コミットの種類を示すキーワード。以下のいずれかを使用すること:
    - `feat`: 新機能の追加
    - `fix`: バグ修正
    - `docs`: ドキュメントの変更
    - `style`: フォーマットの変更（コードの動作に影響しない）
    - `refactor`: リファクタリング（機能追加やバグ修正を伴わないコード変更）
    - `test`: テストの追加・修正
    - `chore`: その他の雑務（ビルドプロセスの変更、依存関係の更新など）
  - `<スコープ>`: 変更が影響するモジュールやコンポーネントの名前（オプション）。
  - `<簡潔な説明>`: 変更内容を簡潔に説明する短い文。50文字以内を推奨。
  - `<詳細な説明>`: 変更内容の詳細な説明。必要に応じて複数行にわたって記述可能。
  - `<関連するIssue番号>`: 関連するIssueの番号を記載する場合は、`#Issue番号`の形式で記述すること。

- **例:**
  - 新機能の追加:

    ```
    feat(auth): add JWT authentication

    Implemented JWT-based authentication for secure user login and session management.

    #42
    ```

  - バグ修正:

    ```
    fix(ui): resolve button alignment issue

    Fixed the alignment issue of buttons on the settings page in mobile view.

    #56
    ```

  - ドキュメントの変更:

    ```
    docs(readme): update installation instructions

    Revised the installation section in README to include new dependencies and setup steps.

    #78
    ```

- **ベストプラクティス:**
  - コミットメッセージは英語で記述することを推奨する。
  - メッセージは命令形で書くこと（例: "Add feature"、"Fix bug"）。
  - 1つのコミットは1つの目的に集中し、複数の変更を含めないこと。
  - 定期的にコミットを行い、大きな変更を一度にコミットしないこと。
- **ツールの活用:**
  - コミットメッセージのフォーマットを自動化するために、`commitizen`や`husky`などのツールを導入することを検討する。
